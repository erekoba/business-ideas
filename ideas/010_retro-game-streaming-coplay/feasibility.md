# 実現性調査

## 副業実現性

- **必要な週稼働時間**: 15〜25時間（立ち上げ期）／5〜10時間（安定運用期）
- **本業との両立しやすさ**: 低
- **理由**: ハードウェアプロトタイプ開発・ソフトウェア開発・配信者への個別営業・技術サポート対応を同時並行する必要があり、立ち上げ期は週15時間以上の稼働が現実的に必要。特にハードウェアトラブルは非同期対応が難しく、配信者の配信中にリアルタイムサポートを求められるケースが発生しやすいため、本業の就業時間外に突発的な稼働が生じるリスクが高い。副業として「週数時間で回せる」構造には現状なっていない。
- **立ち上げまでの期間**: ハードウェアプロトタイプ完成〜βテスター獲得まで3〜6ヶ月、有償転換まで6〜10ヶ月、初期投資回収まで12〜18ヶ月
- **スケールしやすいか（自動化・外注の余地）**: 低。視聴者向け入力インターフェースや課金フローはソフトウェア的に自動化できる余地があるが、ハードウェアの設置・設定・トラブルシュートは配信者側の物理環境に依存するため標準化が難しく、サポート工数が顧客数に比例して増加する構造。スケールするには外注先または共同創業者の確保が前提となる。

## 副業月収ポテンシャル

- **月3万円は達成できるか**: 難しい — 理由: 数字上は配信者15人獲得で達成可能に見えるが、ハードウェアを伴うプロダクトの開発完成・実績づくり・個別営業をすべて副業の限られた時間でこなすには工数が過大であり、15人獲得までに6〜10ヶ月以上を要することが想定されるため、収益化までの期間が長く途中で頓挫するリスクが高い。
- **月10万円は達成できるか**: 難しい — 理由: 配信者35人以上の安定契約に加え、視聴者課金が活発な中堅〜人気配信者を複数確保する必要があり、Vtuber事務所や大手配信者へのアプローチ・メディアへのPRなど、副業の週数時間では到底カバーできない営業・広報工数が必要。副業フェーズでの達成は現実的でない。
- **月数万円達成の最大の壁**: 顧客獲得（ハードウェアを伴うプロダクトへの配信者の導入ハードルの高さと、副業での営業稼働時間の絶対的不足が複合している）
- **副業として取り組む価値があるか**: 条件付き（ハードウェア開発・ソフトウェア開発の両方のスキルを既に保有しており、かつゲーム実況コミュニティへの既存のコネクションがある場合に限り、検証コストを抑えられるため取り組む価値が生まれる）

## 技術的実現性

- **必要なスキル・知識**: 組み込み・ハードウェア開発（Raspberry Pi・マイコン・コントローラー入力変換基板の設計・実装）、映像キャプチャ・低遅延映像中継技術（WebRTC・NDI等）、Webアプリケーション開発（配信者向けダッシュボード・視聴者向けUIの実装）、決済API連携（Stripe等）、ネットワーク・セキュリティの基礎知識（中継サーバー設計）
- **現在の自分のスキルとのギャップ**: アイデアの出発点がソフトウェアよりもハードウェア寄りであるため、組み込み開発・低遅延映像中継の両方に精通していない場合はギャップが大きい。特に「遅延を視聴者が許容できるレベルに抑えながらコントローラー入力を中継する」技術的難度は高く、既存のリモートデスクトップ系技術の転用ではゲームプレイに耐える品質を出しにくい。
- **使用するツール・プラットフォーム**: Raspberry Pi / Arduino（ハードウェアプロトタイプ）、WebRTC（低遅延映像中継）、OBS連携プラグイン、AWS / GCP（中継・ホスティングサーバー）、Stripe（決済）、Discord / Twitter(X)（コミュニティ・営業チャネル）
- **学習コスト**: 高

## リスクと対策

| リスク | 深刻度（高/中/低） | 対策 |
|--------|-------------------|------|
| 遅延問題により視聴者の操作体験が成立しない（ゲームプレイに耐えるレイテンシが確保できない） | 高 | まず遅延許容度が高いターン制・コマンド入力系レトロゲームに対象を絞り、アクション系は最初から対象外とする。プロトタイプ段階で計測・公開し「何ms以内で動作する」という技術的根拠を先に示す |
| 配信プラットフォーム（YouTube・Twitch等）の規約変更・外部ツール制限により突然サービスが成立しなくなる | 高 | 特定プラットフォームへの依存度を下げ、配信者が独自配信サーバーを使うケースにも対応できる設計にする。規約の定期的なモニタリングを運用ルーティンに組み込む |
| ゲームソフト・ハードウェアメーカーの著作権・利用規約に抵触するリスク（特にレトロゲームの実機使用・映像配信） | 高 | 法律・知財の専門家に事前相談し、「実機を使用した配信」の法的グレーゾーンを把握した上でサービス設計を行う。リスクの高いタイトルは対象外とするガイドラインを設ける |
| ハードウェアの個体差・配信者宅の通信環境差により品質が安定しない | 中 | 動作保証環境を厳格に定義し、対応機器リストを公開する。初期は動作環境を絞り込んで対応範囲を限定することで品質を担保する |
| 競合（既存のリモートプレイ機能・クラウドゲーミングサービス）が同様の機能をより低コストで提供し始める | 中 | 「実機のレトロゲーム体験」というニッチな価値訴求に特化し、競合が踏み込まないセグメントに集中する。配信者コミュニティとの関係性構築を競合優位の軸にする |
| 副業稼働時間の不足によりサポート品質が低下し、初期ユーザーが離脱する | 中 | βテスター数を最大10人に絞り、1人ずつ丁寧にオンボーディングすることでサポート工数を管理する。FAQとセルフ解決ドキュメントの整備を早期に行う |

## 次のアクション（検証ステップ）

- [ ] Raspberry Piとキャプチャボードを用いて「コントローラー入力をインターネット越しに中継し、実機レトロゲームを遠隔操作できる」最小プロトタイプを自作し、遅延を計測・記録する（目安：4〜8週間・費用5万円以内）
- [ ] ゲーム実況系のDiscordサーバーやTwitter(X)のコミュニティで「レトロゲーム実況で視聴者参加型にしたいと思ったことがあるか」をアンケート形式で10〜20人にヒアリングし、ニーズの有無と許容できる料金感を確認する（目安：1〜2週間・費用0円）
- [ ] 著作権・配信プラットフォーム規約に詳しい弁護士またはゲーム法務に実績のある専門家に30分〜1時間の相談を行い、実機ゲームの映像配信・遠隔操作中継サービス提供における法的リスクの概要を把握する（目安：2週間以内・費用3〜5万円）

## 結論

- **副業実現性**: 低
- **技術実現性**: 中
- **総評**: アイデアとしての独自性・市場ニーズの芽は認められるが、副業として成立させるには乗り越えるべきハードルが多層的に存在する。技術面では低遅延中継・ハードウェア制御・Web開発を横断するスキルが必要で学習コストが高く、法務面では著作権・プラットフォーム規約の複合リスクがあり、営業面では配信者への個別アプローチを週数時間でこなす工数的限界がある。これら三つのボトルネックが同時に存在するため、副業の週数時間での収益化は現実的に困難と評価する。ただし「遅延計測プロトタイプの自作」「コミュニティヒアリング」「法務相談」の3ステップを低コストで先に実施することで、撤退判断または本格参入判断を早期に下せる。技術力とゲームコミュニティへのコネクションを既に持つ人物が副業ではなく本業・スタートアップとして取り組む場合は、評価が「中〜高」に上振れる可能性がある。
