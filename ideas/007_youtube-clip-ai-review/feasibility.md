# 実現性調査

## 副業実現性

- **必要な週稼働時間**: 立ち上げ期（1〜2ヶ月）は週15〜20時間、安定稼働後は週3〜5時間
- **本業との両立しやすさ**: 中
- **理由**: 安定稼働後はAIによる自動処理・Stripe自動課金が機能するため、運用負荷は極めて低い。ただし立ち上げ期にWebアプリ構築・API統合・LPデザイン・初期プロモーションが集中するため、週末や深夜を使った集中作業期間が1〜2ヶ月必要になる。本業が繁忙期と重なると進捗が止まるリスクがある。また「スコアが高かったのに審査落ちした」などのクレーム対応は不定期に発生するため、完全放置は難しく最低限の問い合わせ確認（1日10分程度）は継続的に必要。
- **立ち上げまでの期間**: 最短6週間（MVP公開まで）。内訳：要件定義・API検証2週間、フロントエンド＋バックエンド実装2週間、テスト＋LP作成1週間、初期ユーザー無料提供・フィードバック収集1週間。ノーコードツール（Bubble等）を活用しない場合はさらに2〜3週間の余裕を見ること。
- **スケールしやすいか（自動化・外注の余地）**: 高。処理フロー（アップロード→解析→レポート返却→削除）が完全自動化可能な構造であり、件数増加に伴う追加工数はほぼゼロ。顧客対応はFAQ整備とテンプレート返信で対応可。コンテンツ発信（X投稿・事例紹介）のみ継続的な人的作業が残るが、外注・スケジュール予約投稿で代替可能。サーバー費用もVercelやAWSのオートスケールで対応できる。

## 技術的実現性

- **必要なスキル・知識**: フロントエンド開発（Next.js またはBubble等ノーコード）、バックエンドAPI設計（Node.js / Python / FastAPI等）、ファイルアップロード処理（S3等へのマルチパート送信）、OpenAI Whisper API連携、Anthropic Claude Haiku Vision API連携（フレーム抽出・Base64変換含む）、Stripe決済実装、基本的なプロンプトエンジニアリング、セキュリティ設計（自動削除・アクセス制御）
- **現在の自分のスキルとのギャップ**: 最大の技術的ハードルは「動画ファイルから一定間隔でフレームを抽出しBase64変換してVision APIに渡す」パイプライン実装。FFmpegを使えば実現可能だが、サーバーレス環境での動作検証が必要でありセットアップに想定外の時間がかかりやすい。WhisperとClaude単体の利用は難易度が低いが、両APIの結果を統合してスコアリングロジックを設計するプロンプト設計は反復検証が必要。Webアプリ開発経験がある場合はギャップ小、経験が浅い場合はギャップ大（ノーコード活用を推奨）。
- **使用するツール・プラットフォーム**: Vercel（ホスティング）、Next.js or Bubble（フロントエンド）、FastAPI or Next.js API Routes（バックエンド）、AWS S3 or Cloudflare R2（一時ファイル保存）、FFmpeg（フレーム抽出）、OpenAI Whisper API（音声解析）、Anthropic Claude Haiku Vision（映像解析）、Stripe（決済）、Resend or SendGrid（メール通知）
- **学習コスト**: 中。Whisper・Stripe単体は公式ドキュメントが充実しており習得コストは低い。Claude Vision APIへの動画フレーム送信パイプラインとFFmpegのサーバー環境対応が学習コストの大半を占める。既存のWebアプリ開発経験があれば2〜3週間で習得可能なレベル。

## リスクと対策

| リスク | 深刻度（高/中/低） | 対策 |
|--------|-------------------|------|
| AIスコアが高いにもかかわらず実審査で落ちる「偽陽性」事例が発生し、口コミで信頼が一気に毀損される | 高 | サービス規約に「審査通過を保証するものではなく、リスク低減ツールである」と明記。スコアより「改善提案の具体性」で価値を訴求し、保証ではなく参考情報としてのポジショニングを徹底する。初期は「問題なし」判定の基準を厳しめに調整し、偽陽性率を意図的に下げる。 |
| YouTubeが審査基準を非公開のまま変更し、サービスの判定ロジックが陳腐化する | 高 | 定期的（月1回）にYouTubeの公式ヘルプ・審査落ちコミュニティ報告を収集し、プロンプト・スコアリングロジックを更新する運用フローを設ける。サービスの価値軸を「審査通過保証」ではなく「動画の独自性改善サポート」に設定することで、基準変更の影響を吸収しやすくする。 |
| 切り抜きチャンネルそのものがYouTubeの規約強化・収益化要件の変更により市場が縮小する | 中 | 対象を切り抜きチャンネルに限定せず、「再利用コンテンツ系チャンネル全般（ゲーム実況・解説・まとめ系）」に拡張できる設計にしておく。初期から特定ジャンルに過度に依存しないLP訴求にする。 |
| 動画アップロード方式による大容量ファイルのサーバー負荷・コスト増 | 中 | アップロードサイズ上限（例：500MB・30分以内）を設定。S3署名付きURLを使った直接アップロード方式でサーバー経由を回避し、コスト・負荷を最小化する。ストレージ一時利用は5分自動削除を徹底する。 |
| 競合（類似ツール）が大手SaaSや無料ツールとして登場し価格競争になる | 低 | 日本語・切り抜き文化特化という文脈と、審査通過実績のコミュニティ形成（口コミ資産）が参入障壁になる。早期に審査通過事例を10〜20件公開し、「実績あるサービス」としてのポジションを先占する。 |

## 結論

- **副業実現性**: 中
- **技術実現性**: 中
- **総評**: 安定稼働後の運用負荷が低く損益分岐点も月18件と現実的であるため、副業ツールとしての構造は優れている。一方で立ち上げ期の技術実装（動画パイプライン・API統合）に2〜6週間の集中投資が必要であり、ここでの挫折リスクが最大の関門となる。技術的には既存APIの組み合わせで実現可能な範囲に収まっており、特別な研究開発は不要だが、「FFmpegのサーバー環境対応」と「スコアリングプロンプトの精度チューニング」の2点で想定外の工数が発生しやすい点は過小評価しないこと。最も重要なリスクは技術ではなく信頼設計であり、「AIスコアが高いのに審査落ちした」という事例をゼロにできない構造である以上、サービスのポジショニングを「審査通過ツール」ではなく「独自性改善ツール」として一貫させることが長期的な事業継続の核心となる。まず技術検証（APIコスト実測）と需要検証（コミュニティアンケート）を並行して1〜2週間で完了させ、どちらかが想定を大きく下回った場合に方向修正する判断軸を早期に持つことを推奨する。
