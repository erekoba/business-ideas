# 実現性調査

## 副業実現性

- **必要な週稼働時間**: 立ち上げ期（1〜2ヶ月）：週10〜15時間／安定運用期：週3〜5時間
- **本業との両立しやすさ**: 中
- **理由**: 運用期に入れば週3〜5時間程度に収束し、非同期作業が中心のため本業との干渉は少ない。ただし立ち上げ期にMVP開発・SNS発信・データ収集を並走させる必要があり、初期2ヶ月は週10時間超の稼働が現実的に求められる。また「予測精度の継続的な改善」はシステム任せにできない部分があり、ゲームリリース直後の集中作業が断続的に発生する点が両立の障壁になり得る。
- **立ち上げまでの期間**: MVP公開まで4〜8週間（NoCode活用の場合4週間、Next.js自作の場合6〜8週間）。収益が安定する目標シナリオ達成までは6〜12ヶ月を想定。
- **スケールしやすいか（自動化・外注の余地）**: 中。Steam API・IGDBからのデータ取得と診断スコア算出はほぼ自動化可能。ただしインディーゲームや国内特有タイトルへの対応・予測モデルの精度チューニング・SNSコンテンツ制作は人手が必要で、外注するにも「ゲームに詳しい人材」というニッチな条件が障壁になる。サブスク1,000人規模を超えた時点でカスタマーサポートの外注を検討する必要が生じる。

## 技術的実現性

- **必要なスキル・知識**: フロントエンド開発（React/Next.js、またはBubble等NoCode）／REST API連携（Steam API・IGDB API）／データベース設計（Supabase等）／サブスク決済実装（Stripe）／アフィリエイトリンク管理／基礎的なデータ分析（プレイ時間・レビュー傾向の数値化ロジック設計）
- **現在の自分のスキルとのギャップ**: Web開発経験がある場合はギャップ小（2〜4週間で着手可能）。未経験の場合はNoCodeツール（Bubble・Glide）を活用すれば開発自体は乗り越えられるが、API連携とStripe実装で躓くリスクが高く、外注費50,000〜150,000円が現実的に必要になる。最大のギャップは「ハマり度を定量スコアに変換するロジック設計」であり、これはツールや外注では代替できない独自の知的設計作業となる。
- **使用するツール・プラットフォーム**: Next.js（またはBubble）／Supabase／Vercel／Stripe／Steam Web API／IGDB API／X（旧Twitter）・Zenn・noteでの情報発信
- **学習コスト**: 中

## リスクと対策

| リスク | 深刻度（高/中/低） | 対策 |
|--------|-------------------|------|
| 「ハマり度予測」の精度が低く、ユーザーの信頼を得られない | 高 | MVP段階ではスコアの根拠（レビュー件数・プレイ時間中央値・返金率など）を透明に開示し、「予測」ではなく「データの可視化」として訴求することで期待値を適切にコントロールする |
| Steam APIの仕様変更・利用規約改定によるサービス停止リスク | 高 | IGDBなど複数APIをバックアップとして並行活用し、特定APIへの依存度を下げる。またゲームDB以外にユーザーの自己申告データも活用できる設計にしておく |
| 無料ユーザーから有料転換率が1%未満にとどまる | 高 | 無料プランで「診断できるが結果が見えない」設計は離脱を招く。無料で1つの軸スコアを開示し、プレミアムで複数軸・比較履歴・類似ユーザー傾向を解放するという明確な価値差分を設計する |
| ゲームメーカー・パブリッシャーからのクレームや法的リスク | 中 | ゲームの画像・動画をスクレイピングで無断使用しない。Steam API・IGDB APIの公式利用規約の範囲内で取得したデータのみを使用し、著作物の引用には出典を明記する |
| 競合（Steamの「あなたへのおすすめ」機能やHowLongToBeat）に機能面で追いつかれる | 中 | Steamは「過去プレイ傾向」ベース、HowLongToBeatは「クリア時間」特化であり、「購入前の自分のハマり度診断」という切り口は差別化可能。ニッチなポジションを言語化してSEO・SNS発信に一貫して活用する |
| 副業時間の確保が困難になりサービス更新が止まる | 低 | 更新頻度が信頼に直結するサービスのため、週1回の定期更新をルール化する。更新停止が2週間を超える場合はユーザーへの告知を必須とし、無言放置によるチャーン（解約）を防ぐ |

## 次のアクション（検証ステップ）

- [ ] Steam APIとIGDB APIのサンドボックスで実際にデータ取得を試し、「ハマり度スコア」の算出ロジックをスプレッドシート上で手動プロトタイプとして作成する（目安：1週間以内）
- [ ] X（旧Twitter）またはnoteに「ゲーム購入前のハマり度を数値化する診断ツールを作っています」という趣旨の投稿を行い、10件以上のリアクション・コメントを収集して需要と懸念点を直接確認する（目安：3日以内）
- [ ] 既存サービス（HowLongToBeat・Backloggd・Steamのレビューページ）を実際に使い込み、「このツールでは解決できていない購入前の不安」を5つ以上書き出してコンセプトの差別化ポイントを言語化する（目安：1週間以内）

## 結論

- **副業実現性**: 中
- **技術実現性**: 中
- **総評**: 技術スタックは標準的なWebアプリの範囲内に収まり、コスト構造も軽量なため「作ること」自体のハードルは中程度と評価できる。しかし本サービスの本質的な価値である「ハマり度の予測精度」は、APIからデータを取得するだけでは実現できず、独自のスコアリングロジックの設計・継続的な改善が必要であり、ここが最大の差別化要因であると同時に最大の難所でもある。副業という時間制約の中でその精度を磨き続けながらSNS発信・サブスク獲得を並走させる負荷は決して軽くない。まずはスコアロジックの手動プロトタイプを作り、SNSで10〜20人のゲーマーに使ってもらって「これは課金したい」という反応が得られるかどうかを2〜4週間で検証することが、最初にして最重要の関門となる。
