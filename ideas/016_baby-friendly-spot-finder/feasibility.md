# 実現性調査

## 副業実現性

- **必要な週稼働時間**: 立ち上げ期（0〜3ヶ月）：週10〜15時間／運用安定期（4ヶ月以降）：週5〜8時間
- **本業との両立しやすさ**: 低
- **理由**: 立ち上げ期はデータ収集・LP構築・SNS運用・施設への個別営業を並行して行う必要があり、週5時間程度では進行速度が極めて遅い。特に施設への法人営業は平日昼間の連絡が必要なケースが多く、本業との時間帯の衝突が生じやすい。また「ユーザーを先に集める→施設を後から課金する」という二段階の構造上、成果が出るまでの期間が長く、モチベーション維持も課題になる。
- **立ち上げまでの期間**: MVP（最小限のサービス公開）まで2〜3ヶ月、初収益発生まで6〜12ヶ月
- **スケールしやすいか（自動化・外注の余地）**: 低。施設情報の収集・更新は人手依存が強く自動化が難しい。法人営業も属人的作業のため外注コストが高い。ユーザー向けコンテンツ（SNS投稿・SEO記事）は外注可能だが、初期収益がない段階での外注費捻出が困難。施設数が増えれば管理工数も線形に増加するため、スケーラビリティは構造的に低い。

## 副業月収ポテンシャル

- **月3万円は達成できるか**: 難しい — 理由: MAU数千人規模のユーザー基盤を先に作らなければ施設側への掲載提案が成立しない。SNS・SEOでそのレベルに達するまで3〜6ヶ月以上を要する見込みであり、その間は実質無収入。その後に6件以上の施設有料契約を副業の稼働時間内で獲得・維持するのは現実的に困難。達成できたとしても立ち上げから9〜15ヶ月以上かかる可能性が高い。
- **月10万円は達成できるか**: 難しい — 理由: 掲載施設20件以上＋広告主2〜3社の並行維持が必要。法人営業・契約管理・ユーザー対応を週5〜8時間で回すことはほぼ不可能。副業単独ではなく、本業化または共同創業者の確保なしに達成できるラインではない。
- **月数万円達成の最大の壁**: 顧客獲得（法人営業の稼働時間不足）。ユーザーと施設の双方を同時に獲得しなければならない二重の獲得コスト構造が根本的な障壁。
- **副業として取り組む価値があるか**: 条件付き — 将来的な本業化・売却・社会貢献を主目的とし、短期収益を期待しないことを前提とする場合に限り意義がある。純粋な副業収益目的であれば費用対効果は低い。

## 技術的実現性

- **必要なスキル・知識**: ノーコードツール（Glide・Bubble・STUDIOなど）またはWeb開発の基礎知識、Google Maps Platform APIの利用設定、施設データのスプレッドシート管理、SNS運用（Instagram・X）、SEOの基礎、法人向け営業・提案資料作成
- **現在の自分のスキルとのギャップ**: Web開発・ノーコード経験がある場合はギャップ小。ない場合はLP構築だけで1〜2ヶ月の学習が必要。地図API連携・施設フィルタリング機能の実装は技術的難度がやや高く、ノーコードでも設計スキルを要する。法人営業の経験がない場合は提案・クロージングのスキル習得が別途必要。
- **使用するツール・プラットフォーム**: STUDIO / Bubble（LP・Webアプリ）、Google Maps Platform（地図・施設表示）、Googleスプレッドシート（施設データ管理）、Notion（運用管理）、Instagram・X（ユーザー獲得）、ChatGPT（コンテンツ作成補助）
- **学習コスト**: 中

## リスクと対策

| リスク | 深刻度（高/中/低） | 対策 |
|--------|-------------------|------|
| ユーザーが集まらず施設営業が成立しない（鶏と卵問題） | 高 | まずInstagramでターゲット層（子育て中の親）に刺さるコンテンツを無料発信し、フォロワー1,000人を先行獲得してから施設営業を開始する順序を徹底する |
| 施設情報の鮮度が維持できずユーザー離れが起きる | 高 | 初期は対象エリアを1〜2都市に絞り込み、更新範囲を限定することで管理工数を圧縮する。ユーザーからの口コミ報告機能を設けてデータ更新を一部クラウドソーシング化する |
| 施設側が無料情報として期待し有料化を拒否される | 中 | 掲載施設に対して「掲載無料・優先表示のみ有料」という段階的な課金設計を採用し、まず無料登録で施設側との関係構築を優先する |
| 大手情報サービス（じゃらん・Googleマップ等）が機能追加で代替してしまう | 中 | 乳幼児特化の詳細フィルター（授乳室の広さ・おむつ台の数・ベビーカー入場可否など）に特化することで差別化領域を確保する |
| 副業稼働時間の不足によりサービス品質が低下する | 中 | 立ち上げ前に週あたりの稼働時間上限を明確に設定し、対象エリア・機能・施設数を最初から絞り込んだMVPに限定してスコープ管理を徹底する |
| 個人情報・位置情報の取り扱いに関する法的リスク | 低 | ユーザーの現在地情報は取得のみでサーバー保存しない設計にし、プライバシーポリシーを弁護士監修または法律テンプレートで整備する |

- [ ] STUDIOまたはGoogleサイトを使い施設リストを掲載したLP（1ページ）を無料で公開し、知人・SNSフォロワー10名以上に見せてフィードバック（使いたいか・何が足りないか）を文字で収集する（目安：1〜2週間）

## 結論

- **副業実現性**: 低
- **技術実現性**: 中
- **総評**: 技術面では、ノーコードツールと地図APIを組み合わせることで、開発スキルが限定的でも最小限のMVPを構築できる可能性がある点は現実的だ。しかし副業としての実現性は低いと判断せざるを得ない。最大の構造的問題は「ユーザー獲得→施設営業→収益化」という三段階のシーケンシャルな成立条件であり、これを副業の週5〜10時間で進めると、初収益まで1年以上かかる公算が高い。社会的ニーズは明確に存在しており、サービスとして成立する可能性はある。ただし、副業として取り組む場合は「収益化よりも市場検証とポートフォリオ構築を目的とする」と割り切り、エリアと機能を最小限に絞ったMVPで需要を先に確認することが唯一の現実的なスタートラインとなる。
