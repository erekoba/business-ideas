# 実現性調査

## 副業実現性

- **必要な週稼働時間**: 20〜30時間（MVP開発フェーズ）／10〜15時間（運用フェーズ）。「週数時間」という副業の一般的な稼働想定を大幅に超過する。
- **本業との両立しやすさ**: 低
- **理由**: 技術開発（クラウドVM設計・SDK開発・決済実装）、配信者への個別営業・関係構築、インディー開発者へのBizDev、運用中のサーバー監視・障害対応という4つの専門領域を同時並行で進める必要がある。それぞれが単独で副業1本分の工数を要するレベルであり、1人が週数時間で全てを回す構造設計になっていない。特に配信者獲得営業はリアルタイムのコミュニケーションと関係構築が必須で、時間を分割して進めにくい性質を持つ。
- **立ち上げまでの期間**: MVPリリースまで最短6〜9ヶ月（自力開発・副業稼働の場合）。収益が副業ラインに達するまでは18〜30ヶ月を要する現実的シナリオが妥当。
- **スケールしやすいか（自動化・外注の余地）**: 低〜中。セッション管理・課金処理・タイマー機能はシステム化により自動化できる。一方、最大のボトルネックである配信者獲得・開発者リレーション構築は自動化が効かない人的工数依存の工程であり、スケールの律速段階になる。SDK導入済みゲームと配信者のエコシステムが自走し始めるまでは外注や自動化の恩恵が出にくい構造。

## 副業月収ポテンシャル

- **月3万円は達成できるか**: 難しい — 理由: 達成条件はアクティブ配信者2〜3名・課金視聴者約67名・SDK対応ゲーム1〜2タイトルの同時確保であり、条件自体は数字上は小さく見える。しかしMVP稼働から最短でも6〜12ヶ月かかり、その間の固定費（サーバー月4〜12万円）が持ち出しになる。収益が黒字化する前にコストが積み上がる構造であり、「副業で月3万円を稼ぐ」という文脈では費用対効果・時間対効果ともに成立しにくい。
- **月10万円は達成できるか**: 難しい — 理由: アクティブ配信者10名以上・課金視聴者200名超・参画ゲーム5タイトル以上の三者同時達成が必要。副業の稼働時間制約のもとで三者エコシステムを同時に成長させることは構造的に困難であり、専業またはチーム体制への移行なしには現実的到達ラインではない。
- **月数万円達成の最大の壁**: 顧客獲得（配信者の確保）。視聴者課金数は配信者の数と影響力に完全依存しており、配信者が0名ではプラットフォームとして機能しない。配信者獲得は時間分割が難しい関係構築業務であり、副業の稼働時間制約と最も相性が悪い。
- **副業として取り組む価値があるか**: 条件付き — 単独副業としては収益性・稼働時間ともに成立しにくいが、「ゲーム×ライブ配信領域でスタートアップ創業を目指す前段のプロトタイプ検証」として位置づけるなら取り組む意義がある。共同創業者（エンジニアまたはBizDev）を確保し、MVP検証までを副業フェーズと割り切る前提であれば条件付きで推奨できる。

## 技術的実現性

- **必要なスキル・知識**: クラウドインフラ設計（AWS/GCP上の仮想マシン管理・ネットワーク設計）、低遅延リアルタイム通信（WebRTC・WebSocket）、セッションハンドオーバー制御ロジックの実装、Unity向けSDK開発（C#）、決済システム連携（Stripe API）、ストリーミング配信プロトコルの理解（RTMP/HLS）、利用規約・契約書の法務知識
- **現在の自分のスキルとのギャップ**: 上記スキルのうち「クラウドVM管理・低遅延リアルタイム通信・SDK開発」の3領域を同時に習得・実装できるエンジニアは少なく、1人が全てをカバーするには相当のギャップが生じる可能性が高い。特に「1秒でセッションを切り替える」というコアUXを支える低遅延制御は、既存のゲームストリーミング技術（NVIDIA GeForce NOW等）のレベルに近い実装難易度を持ち、個人副業レベルで一から構築するには技術ハードルが高い。
- **使用するツール・プラットフォーム**: AWS EC2またはGCP Compute Engine（クラウドVM）、WebRTC（リアルタイム通信）、Unity SDK（インディーゲーム連携）、Stripe（決済）、OBS連携またはカスタム配信クライアント、GitHub（バージョン管理）
- **学習コスト**: 高

## リスクと対策

| リスク | 深刻度（高/中/低） | 対策 |
|--------|-------------------|------|
| 三者（配信者・視聴者・開発者）の同時獲得に失敗し、エコシステムが機能しないまま資金・時間が尽きる | 高 | 最初の3〜6ヶ月は「1配信者×1インディーゲーム×招待制視聴者」の極小クローズドβに絞り、三者同時獲得を最小スコープで成立させる検証から始める。成功事例の動画を残しSNSで拡散することで次の配信者獲得のフックにする |
| 技術実装（低遅延セッション切替）がMVP段階で実現できず、開発が長期化する | 高 | コアUXの遅延許容範囲を初期段階では「3〜5秒」に緩め、WebRTCベースの既存OSS（Parsecのオープン実装等）を活用してスクラッチ開発を避ける。完璧な1秒切替は2〜3フェーズ目の課題と割り切る |
| 既存ゲーム配信プラットフォーム（Twitch・YouTube）が類似機能を内製化し競合優位が消滅する | 高 | インディー開発者向けのSDKエコシステムと収益分配モデルをプラットフォームのロックイン資産として早期に構築する。大手が模倣しにくい「開発者との直接レベニューシェア関係」を差別化軸に据える |
| 粗利率20%の薄利構造のまま固定費が積み上がり、スケール前に資金ショートする | 中 | サーバー費を変動費化（セッション発生時のみVMを起動するサーバーレス設計）し、固定費の下限を月2〜3万円まで圧縮する。あわせて1セッション単価の引き上げ（300〜500円レンジ）を初期ユーザーで検証する |
| 配信者が自身の視聴者をプラットフォームに誘導することに消極的で、集客が起きない | 中 | 配信者にとっての「視聴者エンゲージメント向上」メリットを数値で示す（視聴時間・コメント数の変化）。初期は配信者へのレベニューシェア比率を4割から5〜6割に引き上げ、獲得インセンティブを高める |
| インディーゲーム開発者がSDK導入工数を嫌い参画しない | 低 | Unityアセットストア経由での配布・1クリック導入を実現し、導入工数を最大2時間以下に設計する。初期10タイトルは開発者へ無償サポート付きで導入を支援する |

## 次のアクション（検証ステップ）

- [ ] **技術POCの最小実装（1〜2ヶ月）**: AWS EC2上にゲームVMを1台立て、WebRTCを使って「配信者→視聴者へのキーボード入力ソースの切替」が3秒以内に可能かをローカル環境で検証する。既存OSSライブラリ（Moonlight等）の流用可能性を先に調査し、スクラッチ開発の範囲を最小化する
- [ ] **インディー開発者1名との非公式ヒアリング実施（2週間以内）**: itch.ioまたはUnity Connect上で公開中のインディーゲーム開発者5〜10名にDMを送り、「配信者の配信枠で視聴者に体験プレイさせる機能に対してSDKを導入するか」「収益分配3割で魅力を感じるか」の2点を確認する。商談ではなく課題ヒアリングとして接触コストを下げる
- [ ] **配信者1名とのプロトタイプ共同検証の打診（1ヶ月以内）**: 登録者数1,000〜5,000名規模のインディーゲーム系配信者（YouTubeまたはTwitch）にコンタクトし、クローズドβの共同検証パートナーを1名確保する。報酬はなし・実験的な新機能の先行体験として打診し、合意を得た場合にPOC環境を使ったデモを実施する

## 結論

- **副業実現性**: 低
- **技術実現性**: 低〜中（既存OSSを最大限活用すれば中、スクラッチ前提なら低）
- **総評**: このビジネスアイデアはコンセプトの独自性・市場の方向性・三者レベニューシェアの設計思想いずれも評価できる水準にある。しかし「副業として収益化する」という観点では、技術難易度・三者同時獲得の構造的困難さ・薄利の粗利率という三重の障壁が重なっており、副業単独での収益化は現実的でない。副業として取り組むならば「収益化」ではなく「スタートアップ創業前の仮説検証フェーズ」と定義し直し、技術POCと開発者・配信者ヒアリングに特化した最初の3〜6ヶ月に投資対象を絞るべきである。その検証の結果として「配信者1名・ゲーム1タイトル・視聴者30名」の極小エコシステムが機能することを実証できれば、共同創業者の招集・シード資金調達という次のフェーズへ進む根拠になり得る。現時点で「副業で稼ぐ手段」として評価した場合の結論は「時期尚早」であり、チーム化・資金調達を前提とした起業文脈で再評価することを推奨する。
