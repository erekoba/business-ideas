# 実現性調査

## 副業実現性

- **必要な週稼働時間**: 立ち上げ期（0〜3ヶ月）は週15〜20時間、運用安定期（4ヶ月以降）は週5〜10時間
- **本業との両立しやすさ**: 低
- **理由**: MVP開発フェーズはマルチモーダル解析パイプライン・フロントUI・決済基盤の同時構築が必要であり、週15〜20時間の確保が現実的に困難なケースが多い。特に音声感情解析・顔認識・コメント密度スパイク検出の4要素を統合するパイプライン設計は「週末だけで回せる」レベルを超えており、最低でも平日深夜の継続的な開発時間を要する。運用フェーズ以降はインフラの自動化が機能すれば週5〜10時間程度まで圧縮可能だが、ユーザーサポート・AI精度改善・不具合対応が重なると副業許容範囲を超えるスパイクが頻発するリスクがある。
- **立ち上げまでの期間**: MVPリリースまで3〜5ヶ月。内訳：要件定義・技術選定1ヶ月、パイプライン構築・バックエンド実装1.5〜2ヶ月、フロントUI・決済連携・テスト1〜1.5ヶ月。副業前提で開発速度が本業エンジニアの60〜70%になることを加味した見立て。
- **スケールしやすいか（自動化・外注の余地）**: 動画解析・書き出しのパイプラインはクラウドジョブキュー（AWS SQS等）で自動化しやすく、運用後半はほぼノータッチ運営が可能。ただしAI精度のチューニング・新ゲームタイトルへの対応・顧客サポートは属人的作業が残るため、収益が月50万円を超えた段階でサポート外注（クラウドワークス等で月3〜5万円）を検討すべきフェーズが訪れる。コア技術部分の外注は品質リスクが高く副業段階では非推奨。

## 技術的実現性

- **必要なスキル・知識**: バックエンド開発（Python推奨）、クラウドインフラ構築（AWS or GCP）、動画処理（FFmpeg）、音声認識・感情解析API統合（Whisper / Hume AI等）、顔認識API統合（AWS Rekognition / Azure Face等）、コメントデータ取得・スパイク検出アルゴリズム設計、フロントエンド開発（React or Next.js）、Stripe決済組み込み、CI/CDパイプライン構築
- **現在の自分のスキルとのギャップ**: フルスタック開発経験があればバックエンド・フロント・決済は既存スキルで対応可能。最大のギャップはマルチモーダル解析の統合経験であり、特に「4要素（表情・声・コメント・ゲーム画面）を組み合わせたスコアリングロジック」の設計は実務経験がない場合に試行錯誤コストが大きい。機械学習の研究レベルの知識は不要だが、各APIの精度特性・レイテンシ・コスト構造を理解した上でアーキテクチャを設計できる「AIエンジニアリング」の経験が不足している場合は3〜6ヶ月の学習期間を別途見込む必要がある。
- **使用するツール・プラットフォーム**: OpenAI Whisper（音声文字起こし・感情推定）、Hume AI または AWS Rekognition（表情解析）、FFmpeg（動画切り出し・エンコード）、YouTube Data API / Twitch API（コメントデータ取得）、AWS Lambda + S3 + SQS（サーバーレス動画処理パイプライン）、Next.js + Vercel（フロントUI）、Stripe（決済）、PlanetScale or Supabase（DB）
- **学習コスト**: 高

## リスクと対策

| リスク | 深刻度（高/中/低） | 対策 |
|--------|-------------------|------|
| AI精度が低く「名シーンを外す」検出精度不足により解約が連鎖する | 高 | 最初の3ヶ月はクローズドベータで10〜20名の切り抜き師に無料提供し、精度フィードバックを集中的に反映してからパブリックリリースする。精度が基準（ユーザー満足度70%以上）に達しない場合は公開を延期する判断基準をあらかじめ決めておく |
| 動画解析APIコスト増大による原価割れ（利用量の多いユーザーへの対応） | 高 | 個人プランの月10本上限を技術的にハードキャップとして実装し、超過分は自動的に従量課金（150円/本）に切り替わる仕組みをリリース初日から組み込む。並行してWhisperのセルフホスト化を検討し変動費を30〜50%削減する |
| Opus ClipやVrewが日本語・ゲーム配信特化機能を強化し価格競争に陥る | 高 | 「日本語ゲーム実況特化」の精度優位を定量的に示すベンチマーク比較を公開し差別化を明確化する。コメント欄スパイク検出やゲーム画面UI検知は競合が短期に追いにくい独自技術として先行して磨く |
| 副業稼働時間の不足により開発・運用が停滞する | 中 | MVPのスコープを最小限（音声感情解析＋コメントスパイクの2要素のみ）に絞り、3ヶ月以内にリリースできる設計にする。表情解析・ゲーム画面UI検知はv2以降の機能として切り分ける |
| 著作権・利用規約リスク（ゲーム映像の自動切り抜きに対するゲーム会社・配信プラットフォームの規制強化） | 中 | 利用規約に「素材の権利処理はユーザー責任」と明記し、サービスは技術ツールとして提供する立場を明確化する。弁護士への利用規約レビュー依頼（初期費用3〜5万円）をリリース前に実施する |
| 解約率が月10%を超え顧客基盤が積み上がらない | 中 | 解約時にアンケートを必須化し離脱理由を毎月集計・スプレッドシート管理する。精度起因の解約が30%以上を占めた場合はAI改善にリソースを集中するオペレーションルールを設ける |

## 結論

- **副業実現性**: 低
- **技術実現性**: 中
- **総評**: ビジネスアイデアとしての方向性は正しく、市場ニーズも実在するが、「副業で週数時間」という前提との乖離が最大の課題である。マルチモーダル解析を4要素統合する設計はフルタイムの開発者でも3〜4ヶ月を要する規模であり、副業稼働時間では立ち上げまでに5〜7ヶ月かかる現実的リスクがある。技術実現性は「中」と評価した理由は、各要素単体（Whisper・Rekognition等）は既製APIで実現可能である一方、4要素を統合したスコアリングロジックの精度チューニングが未知数であるためである。推奨する進め方は、まずスコープをWhisper＋コメントスパイクの2要素に絞った「最小MVP」で早期リリースし、精度と需要を検証してから表情解析・ゲーム画面UI検知を追加するフェーズドアプローチである。この方針であれば技術実現性は「高」に引き上がり、副業実現性も「中」まで改善できる可能性がある。
